SKY365 Query Studio
Provider Architecture

A vendor-neutral visual query and agent-planning architecture in which SKY365 owns the semantic query contract, policy enforcement, audit trail, and execution compilers while Radzen and Syncfusion remain replaceable Blazor UI providers.

Approved for implementationDate: 2026-07-17Owner: SKY365 Platform ArchitectureRelated: ADR-AI-007

1. Executive Decision

SKY365 shall not make Radzen, Syncfusion, SQL, OData, or Odoo domains the canonical query model. The platform shall define one SkyQueryPlan contract shared by the natural-language planner, visual Query Studio, policy engine, report builder, dashboard builder, and execution adapters.

Radzen is the default UI provider because it already exists in the SKY365 solution, its Blazor component library is MIT-licensed, and RadzenDataFilter supports grouped filters plus LINQ/OData-oriented output. Syncfusion is the advanced provider for richer nested rule authoring, import/export, parameterized SQL helpers, locking, drag-and-drop scenarios, and its published AI UI Builder workflow.

2. What Radzen Already Provides

  • RadzenDataFilter<TItem> as a visual field/operator/value filter editor with AND/OR grouping.
  • Entity Framework and IQueryable filtering without requiring a custom filter UI.
  • Server-side LoadData integration and OData filtering.
  • Conversion helpers such as ToFilterString and ToODataFilterString.
  • A serializable Radzen.Query model for filter, parameters, ordering, paging, selection, and expansion.
  • Radzen Blazor MCP, which gives coding agents live access to Radzen component documentation, APIs, templates, and usage guidance.
Boundary: Radzen MCP is development-time component knowledge. It is not the SKY365 runtime semantic layer, business vocabulary, authorization model, or query planner.

3. What Syncfusion Adds

  • SfQueryBuilder with structured RuleModel objects and nested rule groups.
  • JSON, SQL, parameterized SQL, named-parameter SQL, and MongoDB import/export capabilities.
  • Programmatic rule locking, group locking, validation, custom value templates, and drag/drop events.
  • A published Blazor UI Builder agent skill using an eight-stage orchestration flow with two decision gates.
  • Progressive disclosure of stage guidance to reduce unnecessary agent context.
Boundary: Syncfusion-generated SQL must never be executed directly as an agent output. SKY365 must validate a canonical plan and compile it server-side under tenant, permission, cost, and policy controls.

  "intent": "Read",
  "entity": "Supplier",
  "select": ["SupplierId", "Name"],
  "metrics": [
    { "name": "TotalPurchases", "aggregation": "sum", "field": "PurchaseAmount" }
  ],
  "filter": {
    "logic": "and",
    "rules": [
      { "field": "DeliveryStatus", "operator": "eq", "value": "Late" },
      { "metric": "TotalPurchases", "operator": "gt", "value": 10000 }
    ]
  },
  "timeRange": { "field": "PurchaseDate", "relativeDays": 90 },
  "sort": [{ "field": "TotalPurchases", "direction": "desc" }],
  "page": { "number": 1, "size": 50 },
  "policyContext": {
    "tenantRequired": true,
    "approvalRequired": false,
    "maxRows": 5000
  }
}

The canonical contract uses business identifiers from the Semantic Catalog. Vendor-specific field names, SQL column names, OData syntax, and Odoo model fields are resolved only inside controlled adapters.

8. Agent Orchestration

Intent and operation classificationDistinguish query, report, dashboard, navigation, draft change, and destructive action before selecting tools.
Project and provider detectionInspect the current SKY365 module, registered component providers, metadata source, and execution target.
Semantic mappingMap natural language to canonical entities, fields, relations, metrics, time semantics, and business statuses.
Query-plan generationCreate SkyQueryPlan; do not generate executable SQL as the source of truth.
Visual projectionRender the plan through Radzen or Syncfusion without losing canonical IDs.
Validation and explanation gateShow interpreted meaning, unresolved terms, risk, estimated scope, and locked policy clauses.
Policy-controlled compilationCompile only after permission, tenant, row-limit, field-sensitivity, and cost checks pass.
Execution and evidenceReturn result, generated artifact, trace, compiler target, policy decision, and audit ID.

9. Safety Rules

  • Read and write plans must use different contracts and tools.
  • Delete, irreversible update, bulk change, and cross-tenant access are never query-builder operations.
  • Mandatory tenant predicates and security filters must be locked and invisible to normal editing where appropriate.
  • Every execution receives row, timeout, join-depth, and cost limits.
  • Unknown semantic terms remain unresolved; they are not guessed into physical columns.
  • Preview and explain are required before high-impact execution.

10. Anti-Goals

  • Do not clone the complete Syncfusion or Radzen designer.
  • Do not expose arbitrary database tables to the agent.
  • Do not store vendor rule models as the only saved format.
  • Do not treat MCP documentation as authorization or runtime policy.
  • Do not promise semantic understanding from UI filtering components alone.
  • Do not couple Report Builder, Dashboard Builder, or Odoo agents directly to a vendor component.

12. Acceptance Criteria

  • A query created in natural language can be opened, edited, and explained visually.
  • A visually created query can be converted back into the same semantic plan without vendor identifiers leaking into the canonical contract.
  • Radzen and Syncfusion adapters pass the same provider contract tests for shared capabilities.
  • Mandatory tenant and permission rules cannot be removed by the user or agent.
  • No generated SQL, OData, or Odoo domain executes before server-side validation and policy evaluation.
  • The platform records provider, plan version, semantic mappings, compiled target, limits, policy decision, and audit identifier.
  • The default deployment can run with Radzen only; advanced deployments may enable Syncfusion without changing stored query definitions.

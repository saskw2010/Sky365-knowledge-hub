ARCHITECTURE DECISION RECORD

Own the Query Plan;
Radzen Default, Syncfusion Advanced

This ADR prevents a Blazor UI component from becoming the semantic source of truth for SKY365 queries, reports, dashboards, and agent actions.

Status: AcceptedDate: 2026-07-17Decision owner: SKY365 Platform ArchitectureBlueprint: BP-AI-007

Context

SKY365 already uses both Radzen and Syncfusion. The platform also needs a visual Query Builder that can represent the same intent created by a user, an AI agent, a report designer, or a dashboard designer. Vendor components provide useful rule-editing capabilities, but they do not own SKY365 business semantics, tenant isolation, permissions, metrics, workflow meaning, destructive-action policy, or audit evidence.

Radzen provides an open-source Blazor component library, RadzenDataFilter, LINQ/OData-oriented conversion helpers, and a documentation MCP service for coding agents. Syncfusion provides a richer SfQueryBuilder, structured nested rules, import/export formats, locking, validation, and a published multi-stage Blazor UI Builder agent skill.

Decision

SKY365 will define and persist a vendor-neutral SkyQueryPlan as the only canonical query contract. Radzen will be the default visual query provider for standard SKY365 screens. Syncfusion will be an optional advanced provider for complex rule authoring and advanced design scenarios. Both providers will communicate through SKY365 adapters and will never execute agent-generated SQL directly.

Decision Details

AreaDecision
Canonical storagePersist SkyQueryPlan, semantic entity IDs, metric IDs, operator IDs, version, and policy metadata.
Default providerUse Radzen for standard filters, grids, saved views, and common ERP query experiences.
Advanced providerEnable Syncfusion for deeply nested rule groups, import/export, rule locking, complex templates, and advanced report/query design.
Agent workflowAdapt the staged orchestration pattern, but keep SKY365 skills, prompts, validators, and design-system rules under SKY365 ownership.
MCP roleUse Radzen MCP and vendor skills as development knowledge sources only; they do not grant runtime authority.
CompilationCompile the canonical plan server-side after semantic, authorization, tenant, cost, and policy checks.
Write operationsKeep create/update/delete commands outside the query-builder contract and route them through typed action contracts and approval rules.

Why Radzen Is the Default

  • It is already part of the SKY365 solution.
  • The runtime component library is open source under the MIT license.
  • RadzenDataFilter covers the standard grouped filtering use case.
  • It integrates naturally with IQueryable, Entity Framework, server-side loading, and OData.
  • It reduces provider cost and lock-in for the default deployment.
Radzen is not being selected as the semantic layer. It is selected as the default visual adapter.

Why Syncfusion Remains Important

  • Its Query Builder exposes a richer structured rule model and advanced manipulation APIs.
  • It supports JSON, SQL, parameterized SQL, named-parameter SQL, and MongoDB rule exchange.
  • It supports locked rules/groups, custom value templates, validation, and advanced interactions.
  • Its Blazor UI Builder demonstrates an eight-stage agent workflow with explicit design and validation gates that can inform SKY365 agent skills.
Syncfusion is not the default source of truth. Advanced capabilities must be mapped into the canonical SKY365 contract.

Rejected Alternatives

AlternativeReason rejected
Use Syncfusion RuleModel as the canonical formatCreates vendor lock-in and mixes UI concerns with semantic and policy concerns.
Use Radzen filter strings as the canonical formatFilter strings are execution-oriented, weaker for metrics, relations, unresolved semantics, policy metadata, and multi-target compilation.
Let the LLM generate SQL directlyUnsafe, difficult to authorize, database-specific, and unable to guarantee tenant and field-level controls.
Build a completely custom visual Query Builder immediatelyDuplicates mature UI work and delays the semantic layer, which is the real differentiator.
Choose one vendor and remove the otherConflicts with the current SKY365 solution and loses useful provider-specific strengths.

Consequences

Positive

  • Natural-language and visual queries share one explainable contract.
  • Radzen and Syncfusion can coexist without fragmenting saved queries.
  • Report Builder, Dashboard Builder, Grid views, and agents reuse the same semantic plan.
  • Provider replacement becomes testable.
  • Authorization and destructive-action protection remain deterministic and vendor-independent.

Costs and Risks

  • Two provider adapters require contract tests and capability negotiation.
  • Some Syncfusion-only features may need graceful degradation in the Radzen adapter.
  • The Semantic Catalog and operator vocabulary must be designed before UI shortcuts become permanent.
  • Saved-plan versioning and migrations become platform responsibilities.
Unspoken failure mode: without a canonical plan, the same business request will produce different meanings in the agent, grid, report, and dashboard layers. A polished Query Builder would then hide semantic inconsistency rather than solve it.

Implementation Constraints

  • Inspect the existing SKY365 code, runtime registration, metadata stores, and database contracts before implementation.
  • Do not assume NoorAutoCrud.razor is the main runtime; current dynamic-screen integration must follow the actual active runtime components.
  • Do not expose physical database fields unless mapped and authorized by the Semantic Catalog.
  • Do not execute SQL or OData emitted by a UI provider without server-side compilation controls.
  • Every query execution must record plan version, semantic mapping, provider, target compiler, policy decision, limits, and audit ID.
  • HTML documentation is mandatory for this decision and its implementation blueprint.

Review Triggers

Review this ADR when any of the following occurs:

  • Radzen publishes a full agent UI Builder equivalent rather than a documentation MCP only.
  • Syncfusion licensing or deployment constraints materially change.
  • SKY365 introduces a custom Query Studio provider.
  • The canonical query plan adds write commands, graph traversal, vector retrieval, or federated multi-source execution.
  • Provider conformance tests reveal material semantic loss.

2. Executive Summary

The session began with a learning question: should SKY365 include a journey that builds a GPT-style model from first principles? The answer is yes, provided the goal is understanding rather than competition with frontier models.

That learning journey was connected to a practical local intelligence architecture. Inspection of the user’s Zvec fork corrected the early idea that Zvec might itself be the agency. Zvec Core is an in-process vector database and retrieval engine. It supplies collections, schemas, vector and text retrieval, filters, indexes, reranking, persistence, and CRUD. It does not supply agent planning, business ontology, a governed semantic layer, or memory lifecycle.

The official ecosystem’s MCP server and Agent Skills are separate projects. The MCP server exposes tools to an external model; the skills package teaches an existing agent how to build with Zvec. Both are deferred until Core understanding, a local proof of concept, stable contracts, and evaluation baselines exist.

Resulting boundary: keep the C++ retrieval core focused; build an idiomatic .NET integration; place Knowledge, Business Semantics, Memory, typed tools, policy, approval, and audit in SKY365-owned layers.

4. Key Decisions — ADRs

ADR-ZVEC-001

Build a Tiny LLM for understanding.

Educational track only; it must feed practical architecture and not become a frontier-model project.

ADR-ZVEC-002

Zvec Core is storage and retrieval infrastructure.

Agent runtime, tools, semantics, memory policy, and safety remain external SKY365 concerns.

ADR-ZVEC-003

Semantic search is not business semantics.

Similarity cannot define authoritative entities, metrics, joins, or operational rules.

ADR-ZVEC-004

Memory management is separate from memory storage.

Zvec may retrieve memories; SKY365 owns lifecycle and governance.

ADR-ZVEC-005

Preserve C++ and build idiomatic .NET.

Native boundary verification is still required before implementation.

ADR-ZVEC-006

Dynamic semantics must be governed.

AI can suggest; approved definitions are human-owned, versioned assets.

ADR-ZVEC-007

Separate knowledge, live tools, behavior, and safety.

Use RAG, typed tools, skills/schemas, and deterministic policy for their proper concerns.

ADR-ZVEC-008

One canonical public hub with separate forks.

Forks are executable research specimens, not the canonical knowledge source.

ADR-ZVEC-009

Defer MCP and Agent Skills.

Do not add orchestration layers before the retrieval foundation is understood and measured.

ADR-ZVEC-010

Inspect first; implement only missing capabilities.

Recover intent and contracts; never port Python line by line without a gap analysis.

ADR-ZVEC-011

Protect the public/private boundary.

No credentials, customer schemas, tenant vocabulary, endpoints, authorization maps, or sensitive rules in public docs.

Reusable Prompt — Upstream Inspection
Confirm fork, upstream, license, branch, and commit relationship. Inventory architecture, dependencies, APIs, tests, persistence, and runtime boundaries. Classify capabilities as implemented, partial, documentation-only, external, or absent. Compare with SKY365. Produce a gap matrix and use/wrap/port/rewrite/reject decision before coding.
Reusable Prompt — C# Tool Reimplementation
Reverse engineer tool names, schemas, validation, lifecycle, state, errors, security assumptions, dependencies, tests, and protocol behavior. Then design an idiomatic .NET implementation. Do not translate line by line.
Reusable Prompt — Governed Semantic Definition
Propose canonical name, bilingual labels and synonyms, sources, grain, identity, relations, calculations, authoritative tool, RAG eligibility, permissions, provenance, validation examples, and unresolved ambiguities. Mark inferred elements as proposals and require approval.
Reusable Workflow — Source Intake
Discover → Classify → Link or fork → Pin commit and license → Inspect → Experiment → Record evidence → Gap matrix → Use/wrap/port/rewrite/reject → Promote canonical decision → Track upstream
Reusable Workflow — Adoption Sequence
Tiny LLM understanding → Zvec Core inspection → Local POC → Retrieval contracts → Native .NET verification → Safe adapter → Knowledge Engine → Governed semantics → Memory lifecycle → MCP → Skills → Agent integration

# SKY365 Unified Daily Sector Intelligence System

**Status:** Canonical operating specification  
**Owner:** Mostafa Elnagar / SKY365 Knowledge Hub  
**Cadence:** Once daily at 08:00 Asia/Kuwait  
**Delivery:** One consolidated report and one email/draft  
**Purpose:** Replace overlapping daily search automations with one governed intelligence cycle organized by sector, evidence, impact, decisions, and actions.

---

## 1. Executive decision

The previous automation model used several independent tasks with overlapping search scopes. That design creates duplication, inconsistent conclusions, fragmented emails, repeated source checks, and no unified decision layer.

The approved model is:

> **One Daily Intelligence Task → multiple sectors → item lists → evidence → impact analysis → decisions → actions → one consolidated report.**

This is not one giant unstructured search prompt. It is a single orchestrated task with explicit internal sectors, source policies, ledgers, deduplication, and decision outputs.

---

## 2. Operating principles

1. Run once each day at 08:00 Asia/Kuwait.
2. Search all sectors in one execution cycle.
3. Use official and primary sources as the source of truth whenever available.
4. Distinguish event date, publication date, page update date, and discovery date.
5. Deduplicate the same development across sectors and sources.
6. Do not repeat unchanged background unless needed to explain a material decision.
7. Canadian immigration draws are never suppressed merely because they do not benefit Safaa.
8. Separate verified fact, secondary reporting, interpretation, and recommendation.
9. Every material item must explain impact on people, business, and planning.
10. Maintain persistent ledgers so each run identifies what is genuinely new.
11. Create one consolidated daily report rather than separate fragmented messages.
12. Preserve direct links and enough source metadata for later verification.

---

## 3. Daily report contract

Each daily report must contain the following top-level sections.

### 3.1 Daily executive brief

- Date and time of run.
- Five most important developments.
- Urgent alerts requiring action today.
- Decisions recommended today.
- Decisions explicitly deferred because evidence is insufficient.
- Items that changed since the previous report.

### 3.2 Cross-sector decision matrix

For every high-value item include:

| Field | Required output |
|---|---|
| Development | What happened |
| Evidence status | Official / primary / confirmed secondary / awaiting confirmation |
| Confidence | High / medium / low |
| Individual impact | Safaa, Reem, Abdulrahman, Mostafa, or other named family members |
| Business impact | WytSky, SKY365, products, sales, funding, delivery, positioning |
| Planning impact | Immigration, relocation, education, hiring, roadmap, budget, timing |
| Urgency | Today / this week / this month / monitor |
| Recommended decision | Act / prepare / validate / defer / ignore |
| Next action | Concrete owner and action |

### 3.3 Sector reports

Each sector follows the same internal structure:

1. **New items**
2. **Material changes to known items**
3. **No-change checks that must be recorded**
4. **Impact analysis**
5. **Decision recommendations**
6. **Action list**
7. **Source ledger update**

### 3.4 Consolidated action register

Classify actions as:

- **Immediate:** action required today.
- **Short term:** action required within seven days.
- **Medium term:** action required within thirty days.
- **Monitor:** no action yet; preserve trigger condition.
- **Blocked:** requires missing evidence, money, document, person, or external decision.

### 3.5 Daily decision log

Record:

- Decision ID.
- Date.
- Sector.
- Decision.
- Evidence used.
- Expected impact.
- Reversal trigger.
- Review date.

## 8. Notification and email behavior

The daily process should generate one concise notification or email/draft containing:

- Executive summary.
- Urgent alerts.
- Top decisions.
- Top actions.
- Sector-by-sector change summary.
- Link or reference to the full Markdown report.

Do not send separate emails for AI, immigration, funding, or repositories.

Exception: an automation that runs only once daily cannot provide true intraday emergency monitoring. Therefore, the accepted trade-off is lower notification noise in exchange for potentially delayed detection until the next 08:00 run. Any sector requiring urgent same-day alerts should later be separated deliberately as a condition watch, not accidentally duplicated.

## 9. Quality gates

Before publishing the daily report verify:

- No duplicated development appears as separate news items.
- Claims are linked to evidence.
- Official sources override secondary interpretation.
- Dates are explicit.
- Safaa impact is not generalized from pharmacist rules to pharmacy-assistant rules.
- Recommendations identify assumptions.
- Business recommendations map to named SKY365 products or capabilities.
- Planning recommendations have a time horizon.
- Actions have an owner or clearly state that ownership is unresolved.
- No destructive repository action is executed.
- No item is marked complete without evidence.

## 11. Architecture decision record

### ADR-DI-001 — Consolidate overlapping daily intelligence automations

**Decision:** Replace independent AI, BC PNP, broad immigration/market, funding, and repository-review daily tasks with one orchestrated daily task.

**Why:**

- Removes duplicated searches.
- Creates one source ledger.
- Produces consistent impact analysis.
- Reduces notification noise.
- Connects external developments to actual decisions.
- Makes GitHub documentation canonical.

**Trade-offs:**

- The prompt is larger and requires strict internal structure.
- A failed run affects all sectors.
- Daily cadence may delay urgent developments.
- Sector token budgets and source depth must be controlled.

**Mitigations:**

- Use persistent ledgers and quality gates.
- Prioritize official sources.
- Limit deep analysis to material changes.
- Keep a future option for a separate true emergency condition watch when justified.

**Status:** Accepted.
